# Teach Concept — 系统化讲课技能

> 把"我不懂"变成"我懂了"的方法论。

## 核心原则

1. **先开黑盒子，再讲代码** — 用户说看不懂，90% 是因为背后有个他不知道的黑盒子（API 的结构、数据的可能性、框架的约定）
2. **等价类覆盖，不留边界** — 每个概念按"正常 → 边界 → 陷阱"三个层次展开，不等人问才补充
3. **一行代码验证一个结论** — 不靠推测靠执行，写一跑一
4. **前端只交可运行代码** — 教学的话终端说；代码文件只留可运行代码和必要注释
5. **先验证再开口** — 任何关于代码行为的断言，必须跑了代码看到结果再说，不靠记忆
6. **诚实退路** — 不确定时先按优先级查资料，不靠猜测回答

## 两阶段教学法

每个概念拆为两个文件，分阶段递进：

### 阶段 1：原子讲解（基础文件）

> 目标：**熟悉用法**

- 每个等价类单独展示，输出直观可见
- 不写函数封装，不引入生产复杂性
- 文件命名：`AXX_ConceptName/XX_ConceptName.py`
- 学生看完知道"哦，原来有这几种写法"

### 阶段 2：应用整合（应用文件）

> 目标：**知识→经验，解决实际问题**

- 写一个可以在项目里直接用的函数/类
- 用真实场景的组合 SQL 演示
- 让学生看到"原来项目里的代码是这么用的"
- 文件命名：`AXX_ConceptName/ConceptName_applied.py`

### 关键规则

- **两个文件都要保留** — 不能删基础文件只留应用版，也不能只有基础没有应用
- **先讲原子，再讲应用** — 学生不懂等价类就直接看生产函数，看不懂
- **应用文件必须可独立运行** — 不依赖基础文件，`python XXX.py` 直接出结果

## 流程图

```
用户说"想学 X" 或 "不懂"
    ↓
[Step 1] 定位黑盒子 — 用户说的这一行/这一段背后依赖什么未知概念？
    ↓
[Step 2] 打开黑盒子 — inspect.getsource 优先，项目内置 wiki/rules 次之
    ↓
[Step 3] 列等价类（教师备课清单，不给用户审批）—
         正常用例 | 边界用例 | 陷阱 | （可选）拓展
    ↓
[Step 4] 逐个等价类讲（从最简实验开始）—
         a) 跑最小实验，看结果
         b) 解释背后的黑盒子
         c) 问"清楚了吗"，确认再进下一个
         d) 在进度地图上标记已覆盖的区域
    ↓ 可能产生新问题超出大纲
[Step 4b] 动态知识点检查 —
         学生提问 → 判定是否新知识点
         ├── 是前置依赖 → 暂停当前概念 → 执行 Step 1~4(新) → 回到原概念
         ├── 是延伸知识 → 标记为"延伸" → 学完当前概念后再覆盖
         └── 非新知识点 → 直接回答后继续
    ↓ 所有等价类覆盖完
[Step 5] 串联 — 用组合例子把所有概念串起来
    ↓
[Step 6] 自动调度 — 根据教学阶段派发子 skill
         ├── 教学过程中学生说"试试/如果改成" → boundary-explore（边界探索）
         ├── Step 5 完成后 → practice-apply（应用实践）
         ├── practice-apply 完成后 → verify-hunch（验证理解）
         └── 任何时候发现新知识点 → 自动写笔记到 00notes/
    ↓ 回到 Step 1，或结束
```

## 流程编排角色

teach-concept 是**总调度者**，不自己做完所有事。以下 3 个 skill 由 teach-concept 在适当时机自动触发：

| skill | 触发时机 | 职责 |
|-------|---------|------|
| `code-reasoning` | 学生说"这段代码看不懂"/"这写的啥" | 三步拆解源码：输入输出→决策树→自动 trace |
| `boundary-explore` | 学生说"试试"/"如果改成"/"猜测"时 | 学生自己写 SQL 验证边界，老师标注差异 |
| `practice-apply` | Step 5 串联完成后 | 写生产级代码，将知识转化为可复用函数 |
| `verify-hunch` | practice-apply 完成后 | 出题/树图对比，验证理解深度 |

**调度规则**：
- 不需要等学生或老师手动调用——teach-concept 在每次问答后判断是否满足触发条件
- 子 skill 完成后自动回到 teach-concept 继续
- 教学过程中产生的疑问 → 写笔记到 `scripts/tutorial/00notes/`（不等提醒）

## 教学辅助工具

讲解时必须使用以下工具辅助可视化，不靠纯文本：

1. **黑板书 `scripts/blackboard_system/blackboard.md`** — 我写 SQL + 标注，学生打开看
2. **AST 树图 `scripts/pglast_debug.py -m 5`** — 解释 SQL 语法结构

## 第 1 步：定位黑盒子

用户说"不懂"，立刻判断：

| 用户的说法 | 黑盒子在哪 | 处理 |
|-----------|-----------|------|
| "这行在做什么" | API 的内部结构或返回值 | dump 类型、属性、可能取值 |
| "为什么这样写" | 设计原因或边界保护 | 列出不同输入的结果对比 |
| "为什么结果不对" | 中间步骤的细节 | 分步打印，对比预期与实际 |

**禁止假设**——不确定时先按 Step 2 的资料优先级查证，不靠记忆回答。

## 第 2 步：打开黑盒子

优先级从高到低：

1. **`inspect.getsource(X)`** — 读类的源码定义，第一时间看到字段和文档注释
2. **查 wiki `.omc/wiki/`** — 项目内部已有记录，避免重复
3. **查 rules `.claude/rules/api/`** — 接口文档和常见操作模式
4. **`scripts/pglast_debug.py`** — 当黑盒子是 SQL AST 结构时使用
5. **`type()` + `dir()` + 遍历实例属性** — 动态看一个对象有什么，最后手段

**只 dump 与当前用户疑问直接相关的属性。** 如果属性太多，先判断用户最可能卡在哪几个，聚焦在这些上。一次性全抛出去等于信息淹没，和不讲差不多。

## 第 3 步：等价类分析（核心）

每个概念必须按以下维度展开：

| 分类 | 含义 | 要求 |
|------|------|------|
| ① 正常用例 | 最常用的写法 | 必须有 |
| ② 边界用例 | 特殊情况（无前缀、星号、空值） | 必须有，要说明预期行为 |
| ③ 陷阱 | 容易踩的坑（会炸的写法、理解误区） | 必须有，说明为什么炸 |
| ④ 拓展 | 很少用但知道了能理解更深的 | 可选 |

每类必须至少有一个可运行的代码例子。

## 第 4 步：逐个等价类讲

不要试图一次性规划全部。从最简实验开始：

- 准备好第一个等价类的实验代码（不写多余代码，不提前写后续等价类的代码）
- 跑代码，展示输出
- 解释背后的黑盒子（打开的具体内容）
- 问"清楚了吗"，确认再进下一个
- 每讲完一个，在进度地图上标记完成

**进度地图必须写入文件**，不能只留在对话上下文里。下次会话加载时会丢失对话中的进度地图。

位置约定：
- 如果教程内容在 `scripts/tutorial/` 下 → 进度地图写入 `scripts/tutorial/INDEX.md`
- 其他场景 → 写入当前学习目录的 `INDEX.md`

进度地图文件内容：
- 当前进度表格（文件/状态）
- 学习路线依赖图

**讲解节奏**：

进度地图格式示例：

```
ColumnRef.fields  ─────────────────
  [✓] 正常: 表名.列名 → ["d","name"]
  [✓] 边界: 裸列名   → ["amount"]
  [ ] 边界: 星号     → 跳过 A_Star
  [ ] 陷阱: 忘记 hasattr 会怎样
```

学生看到时知道"哦，我已经看了前两个，还剩陷阱"，而不是看不懂大纲上的技术名词。

**讲解节奏**：

- 一次只讲一个概念的一个等价类
- 讲完一个等价类问"清楚了吗？"，确认再继续
- 绕开用户能自己看懂的语法细节（如列表推导式、for 循环），只讲黑盒子
- 不行就退：如果当前概念学生卡住了，回到前置概念重讲，不跳过
- **学生说"太快了"必须立刻停**，不要继续赶进度

## 第 4b 步：动态知识点检查

教学过程中学生产生的新疑问可能引出大纲外的知识点。

**判定方法：**

| 疑问类型 | 判定标准 | 操作 |
|---------|---------|------|
| 前置依赖 | 没有这个知识，当前概念无法继续理解 | 暂停 → 学新知识点 → 回到原概念 |
| 延伸知识 | 知道更好，不知道也不影响当前理解 | 标记 → 学完当前概念再覆盖 |
| 普通疑问 | 当前知识点范围内能解答 | 直接回答后继续 |

**前置依赖判断实例：**
```
学 SubLink → 学生问"怎么知道列在子查询里" → 需要 ancestors 知识
判定：Ancestors 是 SubLink 的前置依赖吗？
      → 是的，没有 ancestors 无法理解 visit_SubLink 的第二个参数
      → 暂停 SubLink → 快速讲 Ancestors（in 判断 + depth）→ 回到 SubLink
```

## 决策指南（给等价类附加"什么时候用什么"）

每个等价类讲完后，附加一张"决策指南"表，解决**知识点≠实际操作**的问题：

```markdown
### 决策指南：什么时候用什么

| 场景 | 用什么方法/属性 | 原因 |
|------|---------------|------|
| 判断是否在子查询里 | `ast.SubLink in ancestors` | O(1) 判断，不关心具体嵌套深度 |
| 找到包裹的 SELECT | `ancestors.find_nearest(ast.SelectStmt)` | 从当前节点往上找最近匹配 |
| 知道嵌套深度 | 数 ancestors 中 SelectStmt 的个数 | 多层嵌套时区分内外层 |
| 获取表达式中的列名 | `ColumnExtractor` (Visitor) | 自动走过整个表达式树 |
```

学生看完知道"什么时候该用哪个"，而不是"哦有这么多方法"。

## 讲解技巧（根据实际教学验证）

### ① 用 AST 树图解释结构性问题

当需要解释"为什么这个代码行为不同"时，画树：

```
SelectStmt
  ├── targetList: ResTarget
  │     └── ColumnRef(d.name)  ← 深度 3
  └── fromClause: RangeVar     ← 深度 2 ✅ 先触发
```

```                  
SelectStmt
  ├── targetList: ResTarget
  │     └── ColumnRef(e.name)  ← 深度 3
  └── fromClause: JoinExpr
        ├── RangeVar(employees) ← 深度 3 ❌ 同层！
```

> 树图 + 层数标注，学生一眼看出为什么顺序不同。

### ② 对比教学法：好的 vs 坏的

同时展示"正常的"和"出错的"，问"为什么一个行一个不行"：

- SQL1（简单 SELECT）：RangeVar 在层 2，ColumnRef 在层 3 → ✅
- SQL2（JOIN）：RangeVar 在层 3，ColumnRef 在层 3 → ❌

### ③ 追踪调用链

每行输出必须标注哪个方法产生的：

```python
def visit_ColumnRef(self, ancestors, node):
    print(f'[visit_ColumnRef] 遇到列: {parts}')  # ← 加方法名前缀
    table = self.tables.get(alias, "???")
    print(f'                  查别名={alias} → {table}')  # ← 加缩进区分
```

方法名 + 关键变量值，学生才能对应回代码行。

### ④ 考试笔记 = 速查表

进度地图不只是打勾清单，要包含**可背诵的参考内容**：

```
✅ 节点名 = SQL关键词 + 后缀
   Stmt = 语句 (SelectStmt)
   Expr = 表达式 (A_Expr, BoolExpr)
   Ref  = 引用 (ColumnRef)
   Var  = 变量 (RangeVar)

✅ 属性名 = 英文含义拼接
   rel + name = relname (表名)
   from + Clause = fromClause
   target + List = targetList
```

### ⑤ 代码文件优先，不在终端写临时代码

- 教学代码**必须写入文件**，学生能打开看
- 不要在 `python -c` 里写临时代码——学生看不到，跟丢了就回不来
- 每行输出标注来源方法 + 当前关键变量状态

## 第 5 步：串联

所有等价类讲完后，用组合的例子把前后知识连起来验证理解。

## 先验证再开口（铁律）

任何关于代码行为的断言（"这个类型是 tuple"、"这个节点在子查询里"、"这么写会报错"），遵循以下流程：

```
想说法 → 跑代码验证 → 看到输出 → 再说出口
    ↑              ↓
     ← 如果不对，修正 ←
```

宁愿花 10 秒跑一下，不靠记忆说一句没验证的结论。一次讲错的代价远大于 10 秒的等待。

## 不知道怎么办

当遇到自己不确定的概念时：

1. 诚实说"这个我不确定，让我查一下"——不猜，不编
2. 按 Step 2 的资料优先级查
3. 查到了再讲，查不到如实说："我查了 wiki 和源码都没找到确切说明，你可以上网搜一下 [概念名]，或者我们换个方式来验证这个点"
4. 确定知道的才讲，不确定的等查证后再讲

## 课前备课清单

开讲前老师自己做以下准备（**不抛给学生审批**）：

1. 列出用户疑问背后的原子概念（用 `references/syllabus-template.md` 辅助）
2. 为每个原子概念准备等价类表（正常/边界/陷阱）
3. 按前置依赖排序确认顺序
4. **只准备第一个等价类的实验代码**——不要一次性写完所有代码

## 反模式清单

| 反模式 | 后果 | 正确做法 |
|--------|------|---------|
| 一次写完所有代码再展示 | 用户干等，信息淹没 | 写一行跑一行验证 |
| 不验证就答 | 讲错（如 d.id 子查询问题） | 先跑代码再看结果 |
| 跳过前置概念 | 用户卡住 | 先列出依赖图，不行就退 |
| 只讲正常用例 | 用户遇到边界就懵 | 等价类全覆盖 |
| 代码文件里塞教学评论 | 代码不像代码 | 教学的话终端说 |
| 自己从零写教学代码 | 慢、可能有 bug | 优先用已有工具和脚本 |
| 让学生确认大纲 | 知识诅咒，不会的东西没法审批 | 动态进度地图，讲一段标记一段 |
| 试图一次性规划所有代码 | 写代码时还不知道实际会遇到什么坑 | 只准备第一个等价类的代码 |
| 一行流水账遍历等价类 | 用户看了等于没会，不知道怎么用 | 写一个最小可复用函数，用调用结果展示差异 |
| 进度地图只留在对话中 | 学生/老师回来看不到进度，只能重头再来 | 写入 `scripts/tutorial/INDEX.md` 等持久化文件 |
| 讲解内容写成代码注释 | 代码文件一半是文章，没法当代码用 | 代码只留可运行代码和必要注释，讲解在终端说 |
| 删除基础原子代码只留生产版 | 学生错过分点理解过程，直接面对复杂性 | 基础文件 + 应用文件分开保留 |
| 用 `python -c` 写教学代码 | 学生看不到代码文件，跟丢了回不来 | 所有教学代码写入文件，学生能打开运行 |
| 输出不加方法名前缀 | 学生看到输出不知道哪行代码产生的 | 每行输出标注 `[方法名] + 关键变量` |
| 一次性抛整个文件让学生看 | 学生被信息淹没，不知该看哪几行 | 指出具体行号范围，逐个方法讲解 |
| 同时讲多个概念 | 学生混淆，不知道当前在学什么 | 一个概念讲完、确认、再进下一个 |
| 不使用黑板/AST 树图辅助 | 纯文本讲解，学生看不到 SQL 和树图对应关系 | 必须用 `blackboard.md` + `-m 5` 可视化 |
| 不用黑板只靠嘴讲 | 学生的 IDE 上没内容可看，跟丢了回不来 | 把 SQL 和树图写进黑板，学生打开看 |
| 笔记代替实操 | 学生看了笔记觉得会了，实际跑代码还是卡 | 笔记=知识速查，实操=动手运行；必须让学生跑一遍验证 |

## 核心教学原则：写生产级代码，不写流水账

教程文件里的代码，应该是**你会在实际项目里写的代码**——有函数、有决策逻辑、有可复用的模式。不是一行接一行地打印数据。

```python
# ❌ 反例：流水账 — 看完只会读数据
sqls = [("裸列名", "SELECT name FROM t"), ("表.列", "SELECT d.name FROM t d")]
for label, sql in sqls:
    colref = parse_sql(sql)[0].stmt.targetList[0].val
    parts = [f.sval for f in colref.fields if hasattr(f, 'sval')]
    print(label, parts)
# 学生看完：哦，能提取出 parts... 然后呢？

# ✅ 正例：生产级函数 — 看完可以直接用在项目里
def extract_colref(node: ast.ColumnRef) -> dict:
    """将 ColumnRef 解析为 {alias, column, is_star}"""
    parts = [f.sval for f in node.fields if hasattr(f, 'sval')]
    has_star = any(isinstance(f, ast.A_Star) for f in node.fields)
    if has_star:
        return {"alias": parts[0] if parts else None, "column": None, "is_star": True}
    if not parts:
        return {"alias": None, "column": None, "is_star": True}
    return {
        "alias": parts[0] if len(parts) >= 2 else None,
        "column": parts[-1],
        "is_star": False,
    }

# 然后用一条组合 SQL 展示所有等价类的表现差异
for target in tree[0].stmt.targetList:
    if isinstance(target.val, ast.ColumnRef):
```
